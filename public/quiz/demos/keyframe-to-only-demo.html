👆 Notice anything?
Both boxes animate identically — because the element's own styles already match what an explicit from would have defined. The browser just reads them automatically when from is missing.
The 'to only' keyframe (left box)
/* The element itself defines the start state */
.box-a {
  background: blue;        /* ← browser reads this as 'from' */
  border-radius: 10px;  /* ← and this */
  opacity: 1;           /* ← and this */
  animation: to-only 2s ease-in-out infinite alternate;
}

@keyframes to-only {
  /* No 'from' here — omitted intentionally */
  to {
    transform: rotate(180deg) scale(0.5);
    background: orange;
    border-radius: 50%;
    opacity: 0.3;
  }
}
The explicit from + to version (right box)
.box-b {
  background: green;
  animation: from-and-to 2s ease-in-out infinite alternate;
}

@keyframes from-and-to {
  from {                         /* explicit start */
    transform: rotate(0deg) scale(1);
    background: green;
    border-radius: 10px;
    opacity: 1;
  }
  to {                           /* same end state */
    transform: rotate(180deg) scale(0.5);
    background: orange;
    border-radius: 50%;
    opacity: 0.3;
  }
}

Fade-up notification Element sits at opacity: 0; translateY(20px). Animation only defines to — the hidden state is the natural starting point.
⚠️ One thing to watch out for
If you use animation-fill-mode: backwards or both, the browser applies the from frame during the delay. With no from defined, it just uses the element's existing styles — which is usually exactly what you want anyway.
